> Each app body runs in an opaque-origin sandboxed frame (the API key never enters it), the window
> chrome is host-owned, the desktop the user leaves is restored across reloads, and users can create
> and save custom themes that survive reload FOUC-free — all without revealing the on-demand mechanic.

Met. The four pillars (CHROME → SANDBOX → PERSIST → THEME-editor) are implemented in the
dependency-enforced order, integrated through the `DesktopShell` hub, and individually verified.

## Tech Debt (non-blocking, backlogged)

1. **Theme switch reloads the frame instead of in-place re-skin** (P22 / SANDBOX-05 / THEME-09). `SandboxFrame.srcdoc` is memoized on `themeVars`, so switching themes rebuilds the srcdoc and reloads the iframe; stateful generated apps lose in-frame state. The theme is still applied correctly. Fix path: drop `themeVars` from the srcdoc memo deps so the latent `THEME_PUSH` postMessage becomes the live re-skin path. Recommend backlog → next milestone.
