    Step 1 — Read the persisted layout: const raw = await services.settingsStore.readRaw(LAYOUT_KEY). If raw is null or undefined, return early (fresh session, no layout saved).

    Step 2 — Validate: const layout = deserializeLayout(raw). If layout.length === 0, return early (empty or corrupt layout).

    Step 3 — Sort by z ascending: const sorted = [...layout].sort((a, b) => a.z - b.z). Opening in z-ascending order means the highest-z entry is opened last and naturally appears on top.

    Step 4 — Open all windows atomically (before async resolution): declare opened as an array of {appType, title, instanceId}. For each entry in sorted, call windowManagerRef.current.openAt(entry.appType, {title: entry.title, icon: entry.icon}, {x: entry.x, y: entry.y, z: entry.z, minimized: entry.minimized}) and push {appType: entry.appType, title: entry.title, instanceId} into opened.

    Step 5 — Serial resolution loop: for each {appType, title, instanceId} in opened:
      a. Guard: if (!windowManagerRef.current.isOpenByInstance(instanceId)) continue (window was closed before resolution completed)
      b. Resolve the cache key: const intent = await resolveOpenApp(appType) — this derives the cacheKey without producing anything
      c. Pre-check IDB (PERSIST-03 critical path): const stored = await services.registry.get("apps", intent.cacheKey). If stored is undefined or null AND the app is not in the live/session cache — to keep it simple, just check stored; if null, go to placeholder
      d. If stored is non-null: try { const Component = await resolveComponent(instanceId, appType, intent.cacheKey, services); if (!windowManagerRef.current.isOpenByInstance(instanceId)) { evictLiveComponent(instanceId); continue; } storeComponent(instanceId, Component); } catch { go to placeholder (below) }
      e. Placeholder path (evicted or resolve failed): if (!windowManagerRef.current.isOpenByInstance(instanceId)) continue; const Fallback = makeFallback({ needsAuth: false, throttled: false, onConnect: () => setKeyDialogOpen(true), onRetry: () => { const wid = windowManagerRef.current.windows.find(w => w.instanceId === instanceId)?.id; if (wid) handleClose(wid, instanceId); void handleOpenRef.current(appType, title); } }); storeComponent(instanceId, Fallback)

    The serial (1-at-a-time) structure is provided naturally by the for...of loop with await inside — no concurrency primitive needed. This satisfies PERSIST-02 (concurrency-capped, produce gate not triggered).

    CRITICAL: resolveComponent is ONLY called when services.registry.get("apps", intent.cacheKey) returns a non-null value (confirming tier-3 IDB presence). If the app is not in IDB, the placeholder is shown immediately WITHOUT calling resolveComponent, which is the only code path that calls tryAcquire(). This structurally satisfies PERSIST-03.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| IDB settings store → restoreDesktop | Raw JSON from IDB crosses here; treated as untrusted external data |
| windowManager.windows → writeRaw | React state crosses into IDB; must contain only safe geometry (serializeLayout is the boundary enforcer) |
| resolveOpenApp result → services.registry.get | cacheKey is an opaque string derived from appType; registry.get is read-only and safe |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-08 | Tampering | readRaw("windowLayout") → JSON.parse | mitigate | deserializeLayout wraps JSON.parse in try/catch and filters via isLayoutEntry; corrupt or tampered records produce [] (fresh start); no error reaches the UI |
| T-21-09 | Elevation of Privilege | resolveComponent call on evicted app in restore loop | mitigate | services.registry.get("apps", cacheKey) is called before resolveComponent; a null/undefined result triggers the placeholder path without ever reaching tryAcquire() in loader.ts:320 |
| T-21-10 | Information Disclosure | serializeLayout in save effect writes to IDB | mitigate | serializeLayout (from Plan 21-01) maps WindowEntry to LayoutEntry picking only 7 geometric fields; API key, instanceId, transpiledJS, and Component references never appear in the written JSON |
| T-21-11 | Denial of Service | 50+ geometry changes within 300ms fire 50 IDB writes | mitigate | setTimeout/clearTimeout debounce pattern: each new windowManager.windows change cancels the previous timer; only the timer surviving the 300ms quiet period fires writeRaw — 1 write per drag sequence |
| T-21-12 | Tampering | appType/title from IDB layout in onRetry callback | mitigate | handleOpenRef.current(appType, title) calls the standard handleOpen, which sanitizes via sanitizeDisplayName before using the title; appType is passed to resolveOpenApp which derives a cacheKey via registryKey — no exec path treats appType as executable |
</threat_model>
