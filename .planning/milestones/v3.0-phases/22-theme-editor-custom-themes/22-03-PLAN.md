<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User input → IDB key | User-typed theme name reaches writeRaw as part of "custom:<sanitized>" key; must be sanitized first |
| User input → :root CSS | User-typed color value is set directly on document.documentElement.style.setProperty in live preview; must be gated by CSS.supports before IDB write |
| IDB/localStorage read at edit time | When editingName is provided, vars are loaded from IDB/context; a malformed persisted value should not crash the editor |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-07 | Tampering | User input → IDB key via writeRaw | mitigate | sanitizeDisplayName applied before key construction; key prefix is always "custom:" (hardcoded, not user-supplied); existing IDB keys with built-in names are protected by the namespace |
| T-22-08 | Tampering | User CSS value → setProperty + IDB | mitigate | CSS.supports("background", value) gate before any IDB write; rejected values show an error and never reach writeRaw |
| T-22-09 | Information Disclosure | theme name → DOM aria-label | mitigate | sanitizeDisplayName removes banned tokens before any DOM render, preventing hygiene-banned words from leaking into DOM attributes or user-visible copy |
| T-22-10 | Tampering | malformed vars from IDB on edit load | accept | Editor pre-fills from initialVars (passed by caller from context.customThemes); context already skips malformed entries in refreshCustomThemes (Plan 22-01); editor renders empty string for missing keys — CSS.supports gate at save time validates before re-writing |
</threat_model>
