# Milestone v2.0: Vibe OS

**Status:** ✅ SHIPPED 2026-06-26
**Phases:** 14–18
**Total Plans:** 21

## Overview

v2.0 "Vibe OS" transformed the flat storefront into a themeable, multi-window desktop OS experience. Apps open as draggable glass windows sharing one chrome, several at once, managed by a bottom dock with running indicators and a top menu bar. A dock-launched search/launcher panel lets the user describe an app or pick a pre-installed one; results are placed on the desktop as windows via the real produce loop. A switchable, IndexedDB-persisted 4-theme system (Aurora / Aero / Aqua / Noir) re-skins the entire OS — host chrome and every open app — at a click. Produced apps reference the theme CSS-variable contract, so they re-skin automatically on any theme switch. Zero new npm dependencies: all windowing, drag, and theming are hand-rolled on React 19 multi-root + idb.

**Test growth:** 552 (v1.1) → **727** green. All 21/21 requirements satisfied. 5 phases (14–18). Every UI phase verified by viewed screenshots.

## Phases

### Phase 14: Theme Foundation

**Goal:** The Vibe OS theme contract is live — four named themes apply as CSS custom properties on the document root, the active theme persists with no flash on reload, and a backward-compat alias bridge keeps every pre-v2.0 cached app rendering correctly.
**Depends on:** Phase 13 (v1.1 complete)
**Requirements:** THEME-01, THEME-02, THEME-03, THEME-04, THEME-05
**Plans:** 5 plans

Plans:
- [x] 14-01-PLAN.md — Additive registry DB v2→v3 settings store (durable theme-name mirror foundation)
- [x] 14-02-PLAN.md — Alias bridge CSS (pre-v2 apps keep colors) + osTheme storage key
- [x] 14-03-PLAN.md — VibeThemeProvider + VIBE_THEMES contract on document root, dual persistence via injected settings seam
- [x] 14-04-PLAN.md — FOUC script extension + CSP sha256 hash sync (atomic, no-flash first paint)
- [x] 14-05-PLAN.md — ThemeSelector 4-pill switcher in AppBar + switch-path test (temporary home)

**Details:**
- `VibeThemeProvider` nested inside the existing `ThemeProvider` — named-theme CSS-variable contract layers on top without disturbing the 552-test `data-theme` mechanism.
- Settings store on the `openRegistry()` direct path (not widening the Registry StoreName union) — sits outside the cache-eviction surface.
- `setTheme` fires `settingsStore.write(name)` fire-and-forget — UI switch + localStorage write are synchronous; IDB mirror is best-effort.
- FOUC script duplicates VIBE_THEMES values verbatim from VibeThemeProvider.tsx (must run before any module load); CSP hash update in the same atomic commit.
- ThemeSelector holds no local state — active pill derived from context on every render.
- **Completed:** 2026-06-26

### Phase 15: Window Manager

**Goal:** Apps open as independently draggable glass windows — each with z-order/focus/minimize/close lifecycle managed by a single hook — with no root leaks on close and no pointer-event jank during drag.
**Depends on:** Phase 14 (theme CSS vars must be live so WindowFrame chrome can reference them)
**Requirements:** WIN-01, WIN-02, WIN-03, WIN-04, WIN-05
**Plans:** 4 plans

Plans:
- [x] 15-01-PLAN.md — useDrag: pointer-capture drag hook (setPointerCapture + rAF imperative writes + viewport clamp; state only on pointerup)
- [x] 15-02-PLAN.md — useWindowManager: WindowEntry state + bounded zTop z-order + open/focus/minimize/restore/close (close → unmountApp, zero-leak)
- [x] 15-03-PLAN.md — WindowFrame glass chrome + traffic-light titlebar + body mount seam + desktop/window CSS (display:none minimize, isolation:isolate)
- [x] 15-04-PLAN.md — Rewire App.tsx + Marketplace.tsx open flow to windowed WindowFrames on the desktop; end-to-end integration tests (concurrent, close-leak, mid-produce-close, focus, MOD-still-works)

**Details:**
- Key architectural deviation (sound): windows render in-tree as React subtrees (memoized `WindowBody`) rather than separate `createRoot` roots — the detached-root approach caused test-environment hangs (self-updating app fixture spun unthrottled outside `act()` scope). The deviation eliminates the root-leak class and is net-positive; WIN-02 intent (concurrent, independent) is fully met.
- `Marketplace` owns its own `WindowManagerProvider` so bare-`<Marketplace/>` tests keep working; `App.tsx` also mounts one for desktop-level consumers.
- `AppShell` region appears only once the app resolves; in-flight windows show a neutral placeholder.
- Drag: `setPointerCapture` + rAF imperative `transform` writes + `clamp()` in `pointermove` handler (real-time viewport clamping), state only on `pointerup`.
- **Completed:** 2026-06-26

**Key Decisions:**

| Decision | Rationale |
|----------|-----------|
| Windows render in-tree (memoized `WindowBody`) not separate `createRoot` roots | Detached roots ran outside `act()` scope in tests → self-updating fixtures hung; in-tree removes the root-leak class and is net-positive. |
| Theme vars on `document.documentElement` (not React context) | CSS inheritance reaches all separately-mounted app subtrees; FOUC-safe via `localStorage` sync-read in `index.html`. |
| Zero new npm dependencies | Hand-rolled `useDrag` (setPointerCapture + rAF), `VibeThemeProvider`, all windowing/dock/menu-bar. Research verdict: `react-draggable` has React 19 `findDOMNode` breakage; `framer-motion` is 674KB–4.8MB. |
| `setTheme` fire-and-forget to IDB settings store | UI switch + localStorage are synchronous and authoritative; IDB mirror is best-effort. |
| `colorCheck` allows grayscale hex + neutral-alpha shadows | Prevents false positives on legitimate shadow/border conventions in generated UIs. |
| FOUC script duplicates VIBE_THEMES verbatim | Must run before any module load; kept in sync by convention (copied values). CSP hash update is same-commit invariant. |
